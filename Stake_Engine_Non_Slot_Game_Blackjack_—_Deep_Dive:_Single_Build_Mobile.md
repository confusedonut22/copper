# Stake Engine Non-Slot Game (Blackjack) — Deep Dive: Single Build Mobile Optimization

## Executive Summary

Stake Engine is Easygo's Remote Gaming Server (RGS) and full-stack SDK that lets any developer — indie or studio — build, test, and publish casino games directly to Stake's 20 million+ player audience. Critically for your use case, **there is only one build per game** — and this is by design. The platform passes a `?device=mobile` or `?device=desktop` URL parameter to the same static frontend, and the game itself is responsible for detecting viewport dimensions and adapting its layout reactively. The solution to mobile optimization on Stake Engine is not creating a second build; it is implementing the `utils-layout` package's reactive layout system into your single build, so the blackjack table reflows automatically when the window/device changes.[^1][^2][^3]

***

## Platform Architecture Overview

Stake Engine is built on a monorepo Web SDK powered by Svelte 5, PixiJS 8, and an in-house `pixi-svelte` binding. Games are served as static HTML/JS builds hosted on Stake Engine's CDN and communicate with the RGS backend via a set of authenticated API endpoints.[^4][^3][^5]

The core data flow is:

```
RGS → book → bookEvents → bookEventHandlerMap → emitterEvents → Svelte components
```

When a player places a bet, the RGS returns a **book** — a JSON array of sequential `bookEvents` that tells the frontend exactly how to animate the round. This architecture is designed for slot games but is fully open-ended: you can define **any custom event types** for your blackjack game, and the frontend simply needs a matching handler for each event type.[^6][^4]

### Technologies Inside the SDK

|---|---|

## Building a Non-Slot Game (Blackjack)

### What Makes Blackjack Different from Slots

Slot games on Stake Engine are driven by pre-computed lookup tables and compressed simulation archives generated by the Python Math SDK. For blackjack, outcomes depend on real-time player decisions (hit, stand, double, split) — pre-computing every possible branch is theoretically possible but far more complex.[^7][^8]

The officially documented path for non-standard games is the **custom RGS event model**. Every game on Stake Engine uses the same three core API endpoints:[^3]

- `POST /wallet/authenticate` — validates session, returns balance + config
- `POST /play` — initiates a round, debits the bet
- `POST /wallet/end-round` — completes a round and triggers payout

In between `play` and `end-round`, the game can fire `POST /bet/event` to persist intermediate player actions (e.g., "player hit on 14"), which allows round recovery on disconnect. This is how blackjack's multi-step decision flow maps to Stake Engine's round lifecycle.[^3]

### Mapping Blackjack Hands to Book Events

The RGS responds to each `/play` call with a **book** — in the fifty-fifty example, the book is pre-computed and tells the frontend the outcome immediately. For a real-time card game like blackjack, you would define custom event types and emit them as each action occurs. A typical blackjack round event sequence would look like:[^9]

```json
[
  { "index": 0, "type": "dealCards", "playerHand": ["Ah", "7d"], "dealerVisible": "Kc", "dealerHidden": true },
  { "index": 1, "type": "playerHit", "card": "5s", "newTotal": 23 },
  { "index": 2, "type": "playerBust", "finalTotal": 23 },
  { "index": 3, "type": "roundResult", "outcome": "loss", "payout": 0 },
  { "index": 4, "type": "finalWin", "amount": 0 }
]
```

The frontend `bookEventHandlerMap` then registers a handler for each event type — `dealCards`, `playerHit`, `playerBust`, `roundResult`, `finalWin` — and the `playBookEvents()` function iterates them in sequence, firing animations and state updates synchronously or asynchronously.[^6][^4]

### Math SDK Consideration

The Python Math SDK is designed for pre-simulated slot outcomes. For blackjack:[^7]
- If you implement a **deterministic simulation model** (all possible player paths pre-simulated), you can use the Math SDK to generate lookup tables — but the branching factor makes this very large.
- The more practical approach for blackjack is a **real-time backend** (your own game server), which calls the Stake Engine RGS to open and close rounds while your backend handles the card logic. The TypeScript `stake-engine-client` package provides a lightweight alternative to the full Web SDK for exactly this.[^10][^11]

## The Core Mobile Problem — And the Correct Solution

### Why "Only One Build" Is Not Actually a Limitation

Stake Engine's architecture deliberately serves a **single static build** to all devices. The platform URL that launches your game injects `?device=mobile` or `?device=desktop` as a query parameter. However, this parameter is a *hint* to your game — the real responsive logic happens entirely within your frontend via the `utils-layout` package.[^12][^5][^3]

The key insight: **PixiJS does not have automatic layout flow like HTML/CSS**. Every element must be positioned with explicit coordinates. The `utils-layout` package solves this by providing reactive size, device-type, and layout values that update automatically whenever the browser window resizes.[^13][^14]

### The `utils-layout` Package — Deep Dive

Initialize it once per game:

```ts
import { createLayout } from 'utils-layout';

export const { stateLayout, stateLayoutDerived } = createLayout({
  backgroundRatio: {
    normal: 1.777,    // 16:9 landscape (your desktop table view)
    portrait: 0.5625, // 9:16 portrait (mobile vertical)
  },
  mainSizesMap: {
    desktop:  { width: 1920, height: 1080 },
    tablet:   { width: 1920, height: 1920 },
    landscape:{ width: 1920, height: 1080 },
    portrait: { width: 1080, height: 1920 },
  },
});
```



The `mainSizesMap` defines the **design-space dimensions** for each layout type. The PixiJS stage is then scaled uniformly to fit within the actual canvas while preserving those aspect ratios — meaning you design your blackjack table once at 1920×1080 for desktop, and the engine handles the scale-to-fit math.

### Layout Detection Logic

`stateLayoutDerived` exposes reactive functions (not plain values — they recompute on every resize):[^13]

**Ratio detection (canvasRatioType):**

| Type | Condition | Meaning |
|---|---|---|
| `longWidth` | ratio ≥ 1.3 | Desktop / landscape phone |
| `almostSquare` | 0.8–1.3 | Tablet |
| `longHeight` | ratio ≤ 0.8 | Portrait phone |

**Size detection (canvasSizeType, by shortest dimension):**

| Type | Max Size | Example Device |
|---|---|---|
| `smallMobile` | 375 px | iPhone SE |
| `mobile` | 480 px | iPhone XR |
| `tablet` | 820 px | iPad Air |
| `largeTablet` | 1024 px | iPad Pro |
| `desktop` | > 1024 px | Any desktop |

**Derived `layoutType` (the one you'll use most):**

```ts
const layoutType = () => {
  if (canvasRatioType() === 'almostSquare') return 'tablet';
  if (canvasRatioType() === 'longHeight')   return 'portrait';
  if (canvasSizeType() === 'mobile' || canvasSizeType() === 'smallMobile') return 'landscape';
  return 'desktop';
};
```



### The `isStacked()` Pattern — Your Primary Tool

`isStacked()` returns `true` for both `portrait` and `tablet` layout types. This is the primary branch point for your blackjack game:[^13]

```svelte
{#if context.stateLayoutDerived.isStacked()}
  <!-- Mobile portrait: card areas vertical, buttons at bottom -->
  <BlackjackPortraitLayout />
{:else}
  <!-- Desktop/landscape: traditional horizontal table layout -->
  <BlackjackLandscapeLayout />
{/if}
```

### Responsive Positioning of PixiJS Elements

Because PixiJS uses explicit coordinates, you replace hardcoded pixel positions with reactive canvas-relative values:[^15]

```svelte
<!-- Pin to left edge -->
<Component x={0} />

<!-- Pin to right edge -->
<Component
  x={context.stateLayoutDerived.canvasSizes().width}
  anchor={{ x: 1, y: 0 }}
/>

<!-- Center the main game container (table felt, card areas) -->
<Container
  x={context.stateLayoutDerived.mainLayout().x}
  y={context.stateLayoutDerived.mainLayout().y}
  scale={context.stateLayoutDerived.mainLayout().scale}
/>
```

The `mainLayout()` function returns pre-calculated `{ x, y, scale, width, height, anchor }` so the entire game stage centers and scales automatically.

## Blackjack Mobile Layout Strategy

### The Two-Layout Approach

Given that a blackjack table is fundamentally a **horizontal** design (dealer at top, player at bottom, cards spread side by side), it works naturally in landscape but needs complete reorganization in portrait. The recommended approach within Stake Engine's SDK pattern is to build two Svelte layout components that share the same game state and PixiJS assets, but arrange them differently.[^16][^17]

**Desktop/Landscape Layout (BlackjackLandscapeLayout):**
- Dealer area: top-center
- Player hand area: bottom-center
- Card spread: horizontal
- Action buttons (Hit, Stand, Double, Split): bottom row
- Bet chip selector: bottom-left
- Balance/bet labels: top corners

**Portrait Layout (BlackjackPortraitLayout):**
- Dealer area: top 30% of screen
- Player hand area: center
- Card spread: stacked or fanned vertically
- Action buttons: large, thumb-friendly, bottom 25%
- Bet chip selector: scrollable row above buttons
- Balance/bet labels: top bar

### Implementing the `?device` URL Parameter

While `utils-layout` handles visual responsiveness automatically, you should also read the `?device` URL parameter to apply non-visual optimizations:[^12][^3]

```ts
import { RGSClient } from 'stake-engine';
const client = RGSClient({ url: window.location.href });

// Read device hint
const urlParams = new URLSearchParams(window.location.search);
const deviceHint = urlParams.get('device'); // 'mobile' | 'desktop'

if (deviceHint === 'mobile') {
  // Reduce texture resolution, simplify animations
  // Prefer smaller spritesheets
  // Disable non-essential particle effects
}
```

### UI Components Available from SDK

The `components-ui-pixi` package already ships layout-aware components that the `<UI>` wrapper selects automatically:[^17]

- `LayoutDesktop` — widescreen
- `LayoutLandscape` — landscape phone/tablet
- `LayoutPortrait` — portrait phone
- `LayoutTablet` — tablet aspect ratio

For a blackjack game, **replace the default `<UI>` component entirely** — it is designed for slot games with spin buttons, auto-spin, turbo, etc. Build a custom UI using the raw `stateLayoutDerived` hooks instead. The SDK documentation explicitly states these components are "a starting point, not a final product".[^16][^17]

### Context Setup

Register layout context at your entry component:[^15]

```ts
// context.ts
import { setContextLayout } from 'utils-layout';
import { setContextEventEmitter } from 'utils-event-emitter';
import { setContextXstate } from 'utils-xstate';

export const setContext = () => {
  setContextEventEmitter({ eventEmitter });
  setContextXstate({ stateXstate, stateXstateDerived });
  setContextLayout({ stateLayout, stateLayoutDerived });
};
```

Any descendant component can then call `getContextLayout()` to access reactive layout values without prop-drilling.[^15]

## Mobile Optimization Checklist for Blackjack

### Touch Targets
- Minimum 44×44px touch area for all buttons (Hit, Stand, Double, Split, Deal)
- In portrait mode, action buttons should span most of the screen width for thumb reach
- Use `canvasSizeType()` to scale button sizes up on `smallMobile`

### Asset Strategy
- Use spritesheets for card faces to minimize texture switches[^18]
- Provide `@0.5x.png` resolution-halved sprite variants for mobile — PixiJS will scale them up automatically[^18]
- On `?device=mobile`, consider loading a lower-resolution card spritesheet

### Animation Performance
- Target 60 FPS — disable particle effects and complex win animations on `smallMobile`[^19]
- Card deal animations should be shorter on mobile (less screen real estate to traverse)
- Use `AnimatedSprite` from `pixi-svelte` for frame-based card flip animations[^20]

### SSR Disable
Since PixiJS requires `window`, ensure SSR is disabled in your SvelteKit config:[^21]
```ts
// src/routes/+layout.ts
export const ssr = false;
```

## Build, Deploy, and Testing Workflow

### Single Build Command

```bash
pnpm run build --filter=<your-game-name>
```

Output structure:[^5]
```
apps/<game>/.svelte-kit/output/
  ├── prerendered/pages/index.html
  └── client/  (JS, CSS, assets)
```

Assemble as:
```
build/
  ├── index.html
  ├── _app/
  ├── assets/
  └── favicon.svg
```

### Uploading to Stake Engine

1. Log in to the Stake Engine dashboard
2. Navigate to the **Files** page of your game
3. Click **Import** → select your entire `build/` folder
4. Click **Publish Game** → select **Front End**[^5]

### Testing Mobile vs Desktop in the Same Build

Stake Engine's Developer Testing Tool (launched August 2025) allows instant toggling between device types, currencies, languages, and live vs local builds — without re-uploading. This is the correct workflow for validating your single build across both mobile and desktop before submitting for approval.[^22]

To test locally against the live RGS while hot-reloading:[^5]
1. Launch a game session from the **Developer** page
2. Copy the full query string from the URL (contains `sessionID`, `rgs_url`, `device`, `lang`)
3. Paste into your local dev server URL: `pnpm run dev --filter=<game>`

## Approval Guidelines for Non-Slot Games

### Core Requirements

- Game must compile to a **static build** that functions entirely in the browser[^23]
- All images and fonts must load from the **Stake Engine CDN** — no external asset hosts[^24]
- Game rules and payout information must be accessible from within the UI[^24]
- Math verification is required: RTP targets, probability CSV, lookup tables in the specified format[^25][^23]

### Submission Process

Approval happens entirely through the Stake Engine dashboard:[^23]
1. Navigate to your game's page
2. Click **Request Approval**
3. Complete the form — include a public Google Drive link to supplementary files
4. Stake Engine technical support reviews the submission

For a blackjack game, the math verification will focus on:
- Demonstrating the RTP of your game (standard blackjack with basic strategy is ~99.5%)[^26][^27]
- Showing that the card shuffle and deal logic is provably fair
- Documenting all player decision payouts (hit, stand, split, double, insurance)

### Jurisdiction Considerations

For US-facing play (stake.us social casino), gambling terminology must be adjusted:[^28]
- "BET" → "SPIN" (or in blackjack context, "DEAL")
- "BUY BONUS" → "PLAY BONUS"
- The `stateUrlDerived.social()` check reads the `?social=true` URL param and handles this automatically if you use the SDK's i18n layer[^17]

## Practical Code Pattern: Blackjack Layout Switch

Below is the architectural skeleton for handling the single-build mobile/desktop split in a Stake Engine blackjack game:

```svelte
<!-- BlackjackGame.svelte -->
<script lang="ts">
  import { getContextLayout } from 'utils-layout';
  import BlackjackDesktopTable from './layouts/BlackjackDesktopTable.svelte';
  import BlackjackPortraitTable from './layouts/BlackjackPortraitTable.svelte';

  const { stateLayoutDerived } = getContextLayout();

  // These are FUNCTIONS — they update automatically on resize
  $: layoutType = stateLayoutDerived.layoutType();
  $: isStacked = stateLayoutDerived.isStacked();
  $: canvasW = stateLayoutDerived.canvasSizes().width;
  $: canvasH = stateLayoutDerived.canvasSizes().height;
</script>

<!-- Single conditional — handles all device types in one build -->
{#if isStacked}
  <!-- portrait phone + square tablets -->
  <BlackjackPortraitTable {canvasW} {canvasH} />
{:else}
  <!-- desktop + landscape mobile -->
  <BlackjackDesktopTable {canvasW} {canvasH} />
{/if}
```

Inside each layout component, position PixiJS elements using `mainLayout()` for the table container and `canvasSizes()` for edge-pinned UI elements like balance and bet labels.[^13][^15]

## Key Differences: Slot SDK vs. Blackjack Implementation

| Aspect | Slot Game (SDK Default) | Blackjack (Custom) |
|---|---|---|
| Math backend | Python Math SDK, pre-simulated books | Custom backend OR real-time RGS logic |
| Book events | Pre-defined (reveal, winInfo, setWin) | Custom (dealCards, playerHit, dealerReveal) |
| Player interaction | Single action (spin) | Multi-step decisions per round |
| Round lifecycle | play → end-round automatic | play → /bet/event × N → end-round |
| UI framework | `<UI>` slot game wrapper | Custom UI using raw `stateLayoutDerived` |
| Layout system | `utils-layout` (same) | `utils-layout` (same) |
| Build output | Single static build | Single static build |

## Conclusion and Recommended Action Plan

The "one build only" constraint is not a barrier — it is the intended design. Your desktop blackjack build needs the following additions to work on mobile:

1. **Install `utils-layout`** (already in the web-sdk monorepo as a workspace package)
2. **Initialize `createLayout()`** in your app entry point with both portrait and landscape `mainSizesMap` dimensions
3. **Replace hardcoded PixiJS coordinates** with reactive `canvasSizes()` and `mainLayout()` calls
4. **Add an `isStacked()` branch** to toggle between your existing desktop table component and a new portrait-optimized layout component
5. **Disable SSR** in SvelteKit (`export const ssr = false`)
6. **Test via the Stake Engine Developer Tool** — toggle device types without re-uploading
7. **Optimize assets** — provide `@0.5x` spritesheet variants for mobile card textures

The portrait layout for blackjack is the most work, since the horizontal card table must be rethought vertically. However, the `utils-layout` system handles all the scaling math — you only need to design the component arrangement, not the scaling formulas.
